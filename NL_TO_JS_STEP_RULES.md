# 自然语言转 JS Step 规则简明说明

本文档说明：系统如何把用户写的自然语言步骤转成 `apiGuide.steps`，以及生成后运行时应该选择哪种操作。

## 一句话结论

默认使用本地规则生成 JS steps；只有在 `.env` 设置 `API_GUIDE_USE_LLM=1` 时才启用 LLM 生成。LLM 不可用、超时或返回错误时，会自动回退到本地规则。

重要说明：本地版和 LLM 版只负责“自然语言 -> apiGuide.steps”的生成方式不同。生成完成后，真正执行用例时都走同一个 Midscene JS runner，也就是同一套 `aiTap`、`aiInput`、`aiQuery`、`aiAssert`、cache、重试和截图逻辑。

## 给非技术读者的理解方式

可以把系统想成一个“测试步骤翻译员”。

用户写的是人话，例如：

`在 company code 输入 8540，然后点击 Execute，读取 Curr.bk.val. 列保存为 A1，如果 A1 和 A2 相等则通过`

系统要把它翻译成机器能执行的动作：

1. 找到 company code 字段，输入 8540。
2. 点击 Execute。
3. 去报表表格里读 Curr.bk.val. 这一列。
4. 把读到的值记成 A1。
5. 比较 A1 和 A2 是否相等。

本地规则和 LLM 规则的区别是：

| 方式 | 它像什么 | 怎么工作 |
| --- | --- | --- |
| 本地规则 | 关键词对照表 | 看到固定字眼，就按预先写好的规则转换 |
| LLM 规则 | 给 AI 的操作手册 | 把完整需求交给模型，让模型按手册拆步骤、选动作 |

本地规则更像“如果看到某些关键词，就做某个动作”。LLM 规则更像“给一个懂语言的助手一份说明书，让它判断应该怎么拆”。

当前设计默认优先本地规则，不是因为 LLM 不会做，而是因为 SAP 用例通常步骤长、上下文多。长上下文交给 LLM 时更容易出现耗时久、漏步骤、合并步骤、改写用户原意、输出格式不稳定等问题。本地规则虽然不聪明，但每次结果更一致，更适合稳定执行和缓存复用。

如果已经有 John 给的逐步操作文档，或者有录屏，其实不一定要重新手写一大段自然语言。更快的做法是先用 Claude/LLM 把文档或录屏整理成“每行一个动作”的步骤清单，再交给本地规则或系统生成 apiGuide。这样通常比从零写自然语言更快、更准确。

## 本地规则看到哪些字眼会触发什么动作

本地规则主要靠关键词判断。下面是最重要的触发词。

| 用户自然语言里出现的字眼 | 系统理解成 | 转成的动作 |
| --- | --- | --- |
| `点击`、`选择`、`打开`、`进入`、`返回`、`展开`、`保存` | 要点页面上的东西 | `aiTap` 点击 |
| `勾选`、`启用`、`开启` | 要打开某个选项或复选框 | `aiTap` 点击/勾选 |
| `输入`、`填写`、`填入` | 要往输入框里写值 | `aiInput` 输入 |
| `搜索`、`查询`、`查`，并且后面跟着值 | 多数情况下是往搜索框或字段里输入 | `aiInput` 输入 |
| `回车`、`Enter`、`提交`、`确认`、`Continue`、`Execute`、`执行` | 要按键继续或执行 | `aiKeyboardPress("Enter")`，必要时回退为点击 |
| `Tab` | 要切到下一个字段 | `aiKeyboardPress("Tab")` |
| `F1` 到 `F12` | 要按功能键 | `aiKeyboardPress("F8")` 这类 |
| `读取`、`提取`、`获取`、`查找...值`、`记录为变量` | 要从页面或表格拿数据 | `aiQuery` 读取 |
| `断言`、`验证`、`确认`、`比较`、`对比`、`是否` | 要判断结果对不对 | `aiAssert` 校验 |
| `相等`、`等于`、`不相等`、`不等`、`case通过`、`成功`、`失败` | 要比较变量或判断用例是否通过 | `aiAssert` 校验 |
| `滚动`、`滑动`、`拖动`、`向下`、`向上`、`向左`、`向右` | 要移动页面或表格 | `aiScroll` 或 `aiAct` |
| `等 4s`、`等待 3 秒`、`wait 5s` | 要暂停等待 | `sleep(ms)` |
| `xpath=...` | 用户已经给了精确位置 | 直接用 xpath 定位 |

### SAP 相关触发词

这些词是专门为 SAP WebGUI 加的。

| SAP 写法 | 系统会怎么理解 |
| --- | --- |
| `TC框`、`T-code框`、`事务码框`、`OK Code`、`左上角矩形输入框` | SAP 左上角输入事务码的位置 |
| `company code`、`公司代码` | 公司代码字段 |
| `Asset number`、`资产编号` | 资产编号字段 |
| `Report date`、`报告日期` | 报表日期字段 |
| `List assets` | List assets 复选框 |
| `Execute`、`执行` | 执行查询，通常对应 Execute 按钮、Enter 或 F8 |
| `Favorites` | SAP 首页收藏夹区域 |
| `Menu`、`Settings`、`Visualization` | SAP 菜单设置路径 |
| `列`、`表格`、`记录为变量 A1` | 从 SAP 报表结果表格里取值 |

例子：

| 用户写法 | 本地规则会拆成 |
| --- | --- |
| `TC框搜索 FAGLL03H` | 1. 在 TC 框输入 FAGLL03H；2. 按回车进入 |
| `company code输入8540` | 在 company code 字段输入 8540 |
| `查找 Curr.bk.val. 列的值，记录为变量 A1` | 从报表表格读取 Curr.bk.val. 列，并保存为 A1 |
| `A1和A2相等则case通过` | 比较 A1 和 A2，相等则通过，否则失败 |
| `页面拖动到最右边` | 拖动 SAP 表格底部横向滚动条到最右端 |

## LLM 的操作手册里怎么定义

LLM 版不是简单地问一句“帮我生成代码”。系统会给模型一份很明确的操作手册，让它按规则输出。

这份操作手册可以用普通话理解成下面几条：

1. 你是 SAP WebGUI 自动化测试的步骤规划员。
2. 用户会给你一段自然语言需求。
3. 你只能输出 JSON，不能输出解释、Markdown 或多余文字。
4. 你要把用户的话拆成一步一步的页面操作。
5. 每一步必须说明：第几步、标题、用哪个动作、原始指令、为什么这么选、示例代码。
6. 一步只能做一个动作，不能把多个动作塞进同一步。
7. 一次点击、一次输入、一次勾选、一次执行、一次读取、一次比较，都要分开。
8. 如果一句话里有多个字段输入，例如 company code、Asset number、Report date，要拆成多个输入步骤。
9. SAP 场景要写清楚目标位置，例如 TC/OK Code 框、company code 字段、Execute 按钮、报表列。
10. 点击按钮优先用点击动作。
11. 输入字段优先用输入动作。
12. 提取表格值必须用读取动作。
13. 比较 A1、A2 这类变量必须用断言动作。
14. 不要使用太宽泛的动作，例如“让 AI 自己随便操作整个页面”。
15. 如果缺少账号、密码、目标页面等必要信息，要写到 warnings 里。

### LLM 拿到的 Midscene 函数调用参考

LLM 的操作手册里会告诉它：生成步骤时只能从下面这些常用 Midscene/执行器动作里选，不要自己发明函数。

| 函数/动作 | 什么时候用 | 示例 |
| --- | --- | --- |
| `agent.aiTap("目标")` | 点击按钮、菜单、链接、复选框、图标 | `await agent.aiTap("点击 Execute 按钮");` |
| `agent.aiInput("字段", { value: "值" })` | 给输入框、SAP 字段、搜索框填值 | `await agent.aiInput("company code", { value: "8540" });` |
| `agent.aiKeyboardPress("Enter")` | 按 Enter、Tab、Esc、F8 等键 | `await agent.aiKeyboardPress("Enter");` |
| `agent.aiQuery("string, 要读取的内容")` | 从页面或 SAP 表格里提取值 | `const result = await agent.aiQuery("string, 读取 Curr.bk.val. 列");` |
| `agent.aiAssert("校验条件")` | 判断页面状态或变量比较是否成立 | `await agent.aiAssert("如果 A1 和 A2 相等则通过");` |
| `agent.aiScroll("滚动说明")` | 普通滚动页面或表格 | `await agent.aiScroll("向下滚动一小段");` |
| `agent.aiAct("拖动说明")` | SAP 滚动条拖到最右/最底这类复杂动作 | `await agent.aiAct("拖动底部横向滚动条到最右端");` |
| `sleep(ms)` | 等待页面加载或报表刷新 | `await sleep(4000);` |
| `agent.aiTap("目标", { xpath: "..." })` | 用户已经提供精确 xpath | `await agent.aiTap("点击图标", { xpath: "//*[@id=\"...\"]" });` |

这份参考的作用是限制 LLM 的输出范围：它不能随便写不存在的函数，也不能把所有事情都写成一句“让 AI 操作页面”。它必须把每个动作归类到点击、输入、按键、读取、断言、滚动、等待这些明确类型里。

LLM 每一步要输出的结构大概是：

| 字段 | 给非技术读者的解释 |
| --- | --- |
| `order` | 第几步 |
| `title` | 这一步的短标题 |
| `midsceneApi` | 这一步属于点击、输入、读取、断言还是滚动 |
| `naturalLanguageInstruction` | 真正交给执行器看的那句人话 |
| `reason` | 为什么这一步要用这个动作 |
| `exampleCode` | 系统最终能执行的示例代码 |

LLM 输出后，系统还会检查：

- 它是不是合法 JSON。
- 有没有 steps。
- 每一步是不是只有一个动作。
- 如果模型用了不推荐的宽泛动作，系统会按关键词重新改成更具体的动作。
- 如果 LLM 失败，马上退回本地规则，不让用户卡住。

## 两套规则是怎么制定的

### 本地规则怎么制定

本地规则不是让模型自由理解，而是把 SAP 测试用例中反复出现的写法整理成确定性的转换规则。制定依据主要来自三类内容：

1. 已有 SAP WebGUI 测试用例的自然语言写法。
2. Midscene.js 实际支持且稳定的 API 能力，例如 `aiTap`、`aiInput`、`aiQuery`、`aiAssert`。
3. SAP WebGUI 的特殊操作习惯，例如 OK Code 输入框、Execute、Favorites、报表表格、横向滚动条。

本地规则的制定思路是：

| 规则层级 | 解决什么问题 | 例子 |
| --- | --- | --- |
| 步骤拆分规则 | 把一段话拆成可执行的小动作 | 按 `1.`、逗号、分号、`然后`、`并且` 拆分 |
| SAP 术语标准化 | 把口语写法变成稳定字段描述 | `company code输入8540` -> `在 company code 字段 输入 8540` |
| API 分类规则 | 判断每一步该用哪个 Midscene API | 输入值用 `aiInput`，提取列值用 `aiQuery` |
| 特殊场景规则 | 处理 SAP 特有页面行为 | `TC框搜索FAGLL03H` -> 输入事务码 + 回车 |
| 执行增强规则 | 让生成结果更容易稳定运行 | xpath 直连、变量本地比较、滚动 recovery、cache bypass |

本地规则有明确的 API 选择优先级：

1. 出现断言、验证、比较、相等、case 通过等语义 -> `aiAssert`
2. 出现提取、读取、记录为变量、表格行/列值 -> `aiQuery`
3. 出现字段 + 输入值 -> `aiInput`
4. 出现 Enter、Tab、Esc、F8、继续、执行等按键语义 -> `aiKeyboardPress`
5. 出现点击、选择、勾选、保存、进入、展开等界面动作 -> `aiTap`
6. 出现滚动、滑动、拖动、最右、最底等动作 -> `aiScroll` 或 `aiAct`
7. 识别不了时，默认当成点击动作 -> `aiTap`

本地规则的目标不是“最聪明”，而是“可预测”。同一句自然语言每次生成结果基本一致，便于缓存复用和人工排查。

### LLM 规则怎么制定

LLM 版不是让模型随便输出代码，而是通过 prompt 把它限制成一个“Midscene.js API 步骤规划器”。规则主要写进了模型提示词和返回结果校验里。

LLM 规则的制定依据是：

1. 输出必须是固定 JSON 结构，包含 `summary`、`assumptions`、`steps`、`warnings` 等字段。
2. 每个 step 必须包含 `order`、`title`、`midsceneApi`、`naturalLanguageInstruction`、`reason`、`exampleCode`。
3. 每一步只能选择一个明确 API，不能写 `agent.aiTap() / agent.aiInput()` 这种混合表达。
4. 禁止使用过泛的 `agent.ai()` 和 `agent.aiAct()` 做所有事情，尽量拆成具体 API。
5. SAP 场景必须明确目标位置，例如 TC/OK Code 框、company code 字段、Asset number 字段、Report date 字段、Execute 按钮、指定表格列。
6. 多字段输入必须拆成多步，不能一句话塞进一个 step。
7. 提取变量必须用 `aiQuery`，比较变量必须用 `aiAssert`。
8. 如果 LLM 返回 Markdown 代码块、解释文字或不完整 JSON，系统会尝试清理和校验；失败则回退本地规则。

LLM 版生成后还会做一次“安全归一化”：

- 如果模型返回了 `aiAct` 或 `ai` 这类不推荐 API，会根据指令内容重新推断为更具体的 API。
- 如果 step 没有标题，会用自然语言指令截断生成标题。
- 如果缺少 exampleCode，会按本地规则补一条可执行示例。
- 所有步骤会重新编号，避免模型编号跳号或重复。

所以 LLM 版的目标是“理解更强”，但系统仍然用结构约束和兜底逻辑把它拉回可执行格式。

## 为什么当前优先本地规则

对 SAP WebGUI 测试来说，很多用例不是三五步，而是几十步，里面还会包含字段、表格列、变量、滚动条、设置弹窗、等待时间和 xpath。把整段长上下文直接丢给 LLM，会带来几个问题：

| 问题 | 具体表现 |
| --- | --- |
| 上下文过长 | 模型可能漏掉中间步骤，或者把多个动作合并成一步 |
| 输出不稳定 | 同一段需求多跑几次，拆分结果可能不同 |
| 耗时更久 | LLM 需要完整阅读、规划、输出 JSON，比本地关键词规则慢 |
| 更难复用 cache | 如果模型改写 locator 文案，已有缓存命中率会下降 |
| 容易过度理解 | 用户只是想点击某按钮，模型可能自动补充不存在的前提或步骤 |

所以当前推荐：

1. 标准 SAP 操作、字段输入、表格取值、变量比较：优先本地规则。
2. 本地规则生成后，如果某些步骤明显拆错或语义理解不够，再启用 LLM。
3. 如果有 John 的步骤文档或录屏，优先把它整理成逐步清单，而不是从零写自然语言。
4. 对非常复杂的新流程，可以先用 Claude/LLM 从文档或录屏提取“每一步做什么”，再让系统转成 JS steps。

最理想的输入不是长篇自然语言，而是这种格式：

```text
1. 进入 Menu
2. 点击 Settings
3. 打开 Visualization
4. 勾选 Show OK Code Field
5. 点击 Save
6. 在左上角 OK Code 输入 FAGLL03H
7. 按 Enter
8. 在 company code 输入 8540
9. 点击 Execute
10. 读取 Curr.bk.val. 列，保存为 A1
11. 读取 Book val. 列，保存为 A2
12. 如果 A1 和 A2 相等，则 case 通过
```

这种逐步清单可以来自人工、John 的文档、录屏转写，也可以先让 Claude/LLM 帮忙整理。它比一整段口语描述更适合自动化转换。

## 两套生成规则怎么选

| 场景 | 推荐选择 | 原因 |
| --- | --- | --- |
| SAP 用例步骤比较标准，包含点击、输入、执行、提取、断言 | 本地规则 | 稳定、快、可预测，不容易合并或漏掉步骤 |
| 用例很长，但格式清楚，例如 `1. 2. 3.` 编号步骤 | 本地规则 | 会按编号、逗号、分号、然后、并且拆分成细步骤 |
| 描述比较口语化，需要模型理解上下文和补全拆分 | LLM | 更擅长理解复杂语义和不规则描述 |
| 需要严格复现每一步，不希望模型改写/合并/省略 | 本地规则 | 规则驱动，输出更可控 |
| 本地规则生成后某些步骤分类明显不对 | LLM 或手动编辑 | LLM 可重新规划；也可以直接改 `apiGuide` |

本地规则的优势：

- 不依赖模型配置，速度快，成本低。
- 输出结构稳定，适合 SAP WebGUI 这类重复性强的场景。
- 对常见 SAP 写法有专门处理，例如 `company code输入8540`、`点击 Execute`、`查找某列记录为变量A1`。
- 适合后续缓存复用，因为步骤和 locator 描述更稳定。

LLM 版的优势：

- 更能理解复杂、口语化、跨句子的需求。
- 可以把“启用某设置并保存”这类复合动作拆成更完整的操作链。
- 对没写得很规范的自然语言，容错更好。
- 适合作为本地规则效果不理想时的补充方案。

## 生成链路

1. 用户在 case 里填写 `naturalLanguage`。
2. 点击 `Gen API` 或 `Re-Gen API`。
3. 后端读取 case 中的自然语言。
4. 如果 `API_GUIDE_USE_LLM=1`，先尝试 LLM 生成。
5. 如果未启用 LLM、LLM 未配置或调用失败，使用本地规则生成。
6. 生成结果写回 case 的 `apiGuide`，同时清空旧的 `params` 覆盖值，避免旧参数覆盖新默认值。

## JS Step 操作选择逻辑

| 自然语言意图 | 生成的 JS 操作 | 例子 | 优势 |
| --- | --- | --- | --- |
| 点击按钮、菜单、链接、复选框 | `agent.aiTap()` | 点击 Execute 按钮 | 适合界面点击，表达直接 |
| 往字段里输入值 | `agent.aiInput()` | 在 company code 字段输入 8540 | locator 和 value 分离，便于参数覆盖和缓存复用 |
| Enter、Tab、Esc、F8 等按键 | `agent.aiKeyboardPress()` | 按回车进入事务码 | 对 SAP 快捷键更稳定；无变化时 runner 可回退点击原指令 |
| 提取页面/表格中的值 | `agent.aiQuery()` | 读取 Curr.bk.val. 列，保存为变量 A1 | 返回结构化结果，可保存变量供后续断言使用 |
| 校验页面状态或变量比较 | `agent.aiAssert()` | 如果 A1 和 A2 相等则通过 | 支持本地变量比较，能减少不必要的模型判断 |
| 滚动、拖动页面或表格 | `agent.aiScroll()` / `agent.aiAct()` | 页面拖到最右边 | 对横向/纵向滚动有专门处理，失败时可重试和恢复 |
| 等待页面加载 | `sleep(ms)` | 等 4s | 不走模型，简单稳定 |
| 明确给出 xpath | `agent.aiTap(..., { xpath })` | 点击图标 xpath=... | 直接走 DOM 定位，减少视觉模型误判 |

## 本地规则的关键拆分方式

- 先按编号步骤拆分，例如 `1.`、`2、`。
- 再按 `,`、`;`、`然后`、`并且`、`并` 继续拆分。
- 不会拆开 xpath 里的中括号内容，避免破坏 `//*[@id="M1:46:1[1,0]"]`。
- 会把 SAP 常见简写标准化：
  - `company code输入8540` -> `在 company code 字段 输入 8540`
  - `Asset number输入...` -> `在 Asset number 字段 输入 ...`
  - `Report date输入...` -> `在 Report date 字段 输入 ...`
  - `TC框搜索FAGLL03H` -> 输入事务码 + 回车进入
  - `查找X列的值，记录为变量A1` -> `aiQuery`
  - `A1和A2相等则通过` -> `aiAssert`

## SAP 系统专项规则

这些规则是为了适配 SAP WebGUI 的常见页面结构和操作习惯，不是通用网页自动化规则。

### 事务码和 OK Code

| 用户写法 | 转换结果 | 说明 |
| --- | --- | --- |
| `TC框搜索 S_ALR_87011990` | `aiInput("左上角 TC 框", { value: "S_ALR_87011990" })` + 回车 | 事务码不是点击动作，而是输入动作 |
| `左上角矩形输入框输入 FAGLL03H` | `aiInput("左上角矩形", { value: "FAGLL03H" })` | 兼容 OK Code 输入框没有明确 label 的情况 |
| `按回车进入事务码` | `aiKeyboardPress("Enter")` | SAP WebGUI 里 Enter 常用于进入事务、继续、执行 |
| `点击 Execute` | 优先可转为点击或 Enter 类动作 | Execute 在 SAP 中常等价于 F8/Enter，runner 会做回退处理 |

### SAP 常见字段输入

本地规则会识别常见 SAP 字段，并把“字段名和值混在一起”的写法改成标准 `aiInput(locator, { value })`。

| 用户写法 | 标准化后 |
| --- | --- |
| `company code输入8540` / `公司代码输入8540` | `在 company code 字段 输入 8540` |
| `Asset number输入1010001732` / `资产编号输入1010001732` | `在 Asset number 字段 输入 1010001732` |
| `Report date输入30.04.2026` / `报告日期输入30.04.2026` | `在 Report date 字段 输入 30.04.2026` |
| `posting date开始日期输入01.12.2025` | `aiInput("posting date开始日期", { value: "01.12.2025" })` |
| `search term输入operational division` | `aiInput("search term", { value: "operational division" })` |

这样做的好处是：字段定位描述和输入值分开，后续可以在 Parameters tab 单独改值，不需要改 step，也不容易破坏 cache。

### 菜单和设置项

针对 SAP WebGUI 的菜单设置场景，本地规则会把复合句拆成多个可执行步骤。

例如：

`在 Menu / Settings / Visualization 界面启用 Show OK Code Field 并保存`

会拆成：

1. 点击 `Menu` 菜单
2. 点击 `Settings...`
3. 打开 `Visualization` 设置页
4. 勾选 `Show OK Code Field`
5. 点击 `Save`

这样比一句 `启用 Show OK Code Field 并保存` 更稳定，因为 SAP 弹窗和菜单层级需要逐步定位。

### Favorites 和主页导航

如果自然语言写：

`返回 SAP 主页，选择 Favorites 下面的 Asset Balances`

会拆成：

1. 返回 SAP 主页
2. 展开 `Favorites` 文件夹并选择 `Asset Balances`

如果后面还有 `点击 Execute`、`输入 company code` 等动作，会继续递归拆分，不会把后续动作丢掉。

### 表格取值和变量

SAP 报表结果通常在表格中，因此本地规则对“列值提取”做了专门处理。

| 用户写法 | 转换结果 |
| --- | --- |
| `查找 Curr.bk.val. 列的值，记录为变量 A1` | `aiQuery("string, 从查询结果表格中读取 Curr.bk.val. 列，保存为变量 A1")` |
| `读取 Book val. 列，保存为变量 A2` | `aiQuery(...)` 并在运行时保存变量 |
| `A1 和 A2 相等则 case 通过` | `aiAssert(...)`，优先走本地变量比较 |

运行时如果断言是变量比较，例如 `A1 == A2`，runner 会优先用本地变量 Map 比较，避免每次都交给视觉模型判断。

### SAP 滚动条和横向表格

SAP 报表经常有横向列很多的问题，所以本地规则会识别：

- `页面拖动到最右边`
- `表格滑到最底端`
- `滚动到最左边`
- `向下滚动一小段`

规则：

- “最右/最左/最上/最底”这类极端滚动，会生成更明确的拖动指令，例如拖动底部横向滚动条到最右端。
- 普通滚动会生成小幅滚动动作，失败时 runner 会继续做 scroll recovery。
- 在 cached 模式下，拖动/滚动类 step 默认建议 `bypass cache`，因为 SAP 滚动条位置容易随会话变化。

### xpath 直连规则

如果用户在自然语言末尾写 `xpath=...`，系统会把 xpath 从自然语言中拆出来。

例如：

`点击图标 xpath=//*[@id="C185_toolbar_btn10"]`

会转成：

`await agent.aiTap("点击图标", { xpath: "//*[@id=\"C185_toolbar_btn10\"]" });`

优势：

- 不需要模型根据截图猜图标。
- 直接通过 DOM 定位，速度更快。
- 对 SAP toolbar 图标、无文本按钮更稳定。

### 动态日期

显式日期会原样输入，例如 `30.04.2026`、`2026-04-30`。

如果输入值是动态日期表达式，例如：

- `今天`
- `上个月最后一个自然日`
- `本月月底`
- `last month end`

运行时会尝试解析成 SAP 可输入的具体日期，默认倾向 `DD.MM.YYYY` 格式。这个解析需要 LLM 配置；如果没有配置，建议直接写明确日期。

## 运行操作怎么选

| 操作 | 什么时候用 | 优势 |
| --- | --- | --- |
| `Run raw` / `Run JS` | 第一次运行、新改了 T Code、新改了步骤、页面结构可能变了 | 重新规划 locator，写入新的 cache |
| `Run cached` / `Run JS w/ Cache` | 用例已经成功跑过，想快速稳定复跑 | 复用上次成功运行的定位结果，通常更快、更稳定 |
| `bypass cache` 某一步 | cached 模式下，某一步 locator 可能过期 | 只让这一小步重新问模型，其它步骤继续走缓存 |
| `keep cache even if run fails` | raw 调试阶段，希望保留失败前已写入的定位结果 | 下一次 raw/cached 可复用已有进展，加速调试 |

建议流程：

1. 先写自然语言。
2. 点 `Gen API` 生成 steps。
3. 人工快速检查 `aiInput`、`aiQuery`、`aiAssert` 是否合理。
4. 第一次用 `Run raw`。
5. 成功后再用 `Run cached` 做稳定复跑。
6. 如果 cached 某一步失败，只勾选那一步的 `bypass cache` 后再跑。

## 维护建议

- 自然语言尽量写成一行一个动作，复杂句用编号。
- 输入字段写清楚“字段名 + 值”，例如 `在 company code 字段输入 8540`。
- 提取值时写清楚“哪一列/哪一行 + 保存为哪个变量”。
- 断言写成可观察结果，例如 `如果变量 A1 和 A2 相等，则测试通过`。
- 能提供 xpath 的稳定按钮，可以在自然语言末尾加 `xpath=...`。
