业务用户从打开 Run Center 到拿到结果的 5 个步骤；每步都对应 UI 上的具体元素，没有"再点 6 层菜单"的脑负担。

1

打开 Run Center

原控制台顶 bar 主蓝按钮，或直接走 /run-center.html
看到 11 个 SAP cases + 4 个 category bucket + Saved batches。
2

选 case

勾 checkbox / 用 Select all / chip 筛 AP Report
每个 case 可展开行内改参数 + 折叠看 NL，不必跳出。
3

命名 + 起跑

▶ Run · N cases → 弹 Confirm Modal
输 batch 名字 + 选 ×N → Enter / 点 Start running 启动。
4

实时看转播

Run Console 自动隐藏，Live Takeover 接管
大屏 broadcast + flip chips 切 case + Live Log + Abort 弹窗。
5

复盘 + 复跑

My Projects 卡片每行 Report / Log
挑失败 case → ↺ Rerun，或者 Edit batch 调整后再跑。

关键设计取舍

每条都是用户用过后吐槽 → 我们重新选的方向。把"为什么这么做"显式写出来，便于后续 review 不走回头路。

Why · 误触保护

所有 Run 入口强制走 Confirm Modal

点 Run → 弹窗 → 命名 → 真启动
点 Run → 直接开始跑

跑 11 个 case × ×5 次 = 55 次浏览器拉起，启动成本不可逆。命名让 batch 沉淀成 My Projects 复用资产，不是匿名空跑。

Why · 视觉一致

开发者控制台 ⇄ 用户控制台 双向链接

两个页面顶 bar 互链 + 不同主题
合并成一个页面"模式切换"

开发场景（编辑 case JSON / 调 cache / 看截图）和运行场景（选 case / 跑 / 监控）操作密度差异大。强行合并会两边都做不好。

Why · 实时性 vs 资源

2.5s 轮询 active runs，先不上 WebSocket

setInterval 2500ms + 屏幕指纹比对
WebSocket 实时推

当前 active run 上限 10 个、log 30 行/run = 单次 payload <10KB，2.5s 间隔足够。WebSocket 留到 v3，需要做断线重连 + 鉴权。

Why · 状态机简化

Live Takeover 自动接管，不留"Run + Live"双视图

跑起来 → Console 隐藏 → 自动还原
永远显示 Console + Live 分裂屏

用户启动后 95% 时间在看 broadcast，5% 想加新 case。Compose new run 逃生口照顾后者，主路径不被 Console 干扰。

Why · 分类要"够用"

4 个 category bucket（AP / FI / Verify / Create）

4 个粗 bucket + 颜色 + 重复 label
11 个独立精细 label / 透传 TCODE

11 个 case 不到分 10 类的体量。用户要的是"按业务领域批量跑"，TCODE 是码农工具。重复 label 反而帮用户视觉聚类。

Why · 失败自救

所有 renderXxx 包 try/catch + safeRender

单个 panel 报错只挂自己，不影响 tab/sidebar
一处 throw → 整个页面卡死

localStorage 残留脏数据 / 接口偶发返回奇怪 shape 都不该让全页崩。错误 panel 还会附"清空缓存"按钮让用户自救。

页面结构 / Tab 关系

三个顶部 Tab + 内部 sidebar，按使用频率排列：日常运行走 Run Center，单 case 编辑去 Case Editor，多 case 监视去 Live Channels。
