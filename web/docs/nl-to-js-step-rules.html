自然语言转 JS Step 规则说明

说明系统如何把 SAP 测试用例的自然语言转换成 apiGuide.steps，以及本地规则和 LLM 规则分别怎么制定、怎么选择。

默认使用本地规则生成 JS steps；只有在 .env 设置 API_GUIDE_USE_LLM=1 时才启用 LLM。LLM 不可用、超时或返回错误时，会自动回退到本地规则。 本地版和 LLM 版只负责生成方式不同，最终执行都走同一个 Midscene JS runner。

理解方式

可以把系统想成一个“测试步骤翻译员”。用户写的是人话，系统要把它翻译成机器能执行的动作。

例如用户写：在 company code 输入 8540，然后点击 Execute，读取 Curr.bk.val. 列保存为 A1，如果 A1 和 A2 相等则通过

  1. 找到 company code 字段，输入 8540。
  2. 点击 Execute。
  3. 去报表表格里读 Curr.bk.val. 这一列。
  4. 把读到的值记成 A1。
  5. 比较 A1 和 A2 是否相等。
方式它像什么怎么工作
本地规则关键词对照表看到固定字眼，就按预先写好的规则转换
LLM 规则给 AI 的操作手册把完整需求交给模型，让模型按手册拆步骤、选动作

当前设计默认优先本地规则，不是因为 LLM 不会做，而是因为 SAP 用例通常步骤长、上下文多。长上下文交给 LLM 时更容易出现耗时久、漏步骤、合并步骤、改写用户原意、输出格式不稳定等问题。本地规则虽然不聪明，但每次结果更一致，更适合稳定执行和缓存复用。

如果已经有 John 给的逐步操作文档，或者有录屏，其实不一定要重新手写一大段自然语言。更快的做法是先用 Claude/LLM 把文档或录屏整理成“每行一个动作”的步骤清单，再交给本地规则或系统生成 apiGuide。这样通常比从零写自然语言更快、更准确。

LLM 的操作手册里怎么定义

LLM 版不是简单地问一句“帮我生成代码”。系统会给模型一份明确操作手册，让它按规则输出。

  1. 你是 SAP WebGUI 自动化测试的步骤规划员。
  2. 用户会给你一段自然语言需求。
  3. 你只能输出 JSON，不能输出解释、Markdown 或多余文字。
  4. 你要把用户的话拆成一步一步的页面操作。
  5. 每一步必须说明：第几步、标题、用哪个动作、原始指令、为什么这么选、示例代码。
  6. 一步只能做一个动作，不能把多个动作塞进同一步。
  7. 一次点击、一次输入、一次勾选、一次执行、一次读取、一次比较，都要分开。
  8. 如果一句话里有多个字段输入，例如 company code、Asset number、Report date，要拆成多个输入步骤。
  9. SAP 场景要写清楚目标位置，例如 TC/OK Code 框、company code 字段、Execute 按钮、报表列。
  10. 点击按钮优先用点击动作，输入字段优先用输入动作。
  11. 提取表格值必须用读取动作，比较 A1、A2 这类变量必须用断言动作。
  12. 不要使用太宽泛的动作，例如“让 AI 自己随便操作整个页面”。
  13. 如果缺少账号、密码、目标页面等必要信息，要写到 warnings 里。

为什么当前优先本地规则

对 SAP WebGUI 测试来说，很多用例不是三五步，而是几十步，里面还会包含字段、表格列、变量、滚动条、设置弹窗、等待时间和 xpath。把整段长上下文直接丢给 LLM，会带来几个问题：

问题具体表现
上下文过长模型可能漏掉中间步骤，或者把多个动作合并成一步
输出不稳定同一段需求多跑几次，拆分结果可能不同
耗时更久LLM 需要完整阅读、规划、输出 JSON，比本地关键词规则慢
更难复用 cache如果模型改写 locator 文案，已有缓存命中率会下降
容易过度理解用户只是想点击某按钮，模型可能自动补充不存在的前提或步骤

因此推荐：标准 SAP 操作优先本地；本地拆错时再启用 LLM；如果有 John 的步骤文档或录屏，优先把它整理成逐步清单，而不是从零写自然语言。

最理想的输入是“每行一个动作”的清单，例如：

  1. 进入 Menu
  2. 点击 Settings
  3. 打开 Visualization
  4. 勾选 Show OK Code Field
  5. 点击 Save
  6. 在左上角 OK Code 输入 FAGLL03H
  7. 按 Enter
  8. 在 company code 输入 8540
  9. 点击 Execute
  10. 读取 Curr.bk.val. 列，保存为 A1
  11. 读取 Book val. 列，保存为 A2
  12. 如果 A1 和 A2 相等，则 case 通过

两套规则是怎么制定的

本地规则

本地规则来自已有 SAP WebGUI 测试用例和稳定的 Midscene API 能力，把高频写法沉淀成确定性转换规则。

  • 先拆分步骤：编号、逗号、分号、然后、并且。
  • 再标准化 SAP 术语：OK Code、company code、Asset number、Report date。
  • 再选择 API：输入用 aiInput，点击用 aiTap，取值用 aiQuery，断言用 aiAssert。
  • 最后加执行增强：xpath 直连、变量本地比较、滚动恢复、cache bypass。

目标是可预测、稳定、便于缓存复用。

LLM 规则

LLM 版通过 prompt 把模型限制成 Midscene.js API 步骤规划器，不允许自由输出代码。

  • 输出必须是固定 JSON。
  • 每个 step 必须包含 API、自然语言指令、原因和示例代码。
  • 每一步只能使用一个明确 API。
  • 禁止用泛化的 agent.ai() 代替具体动作。
  • 返回后还会校验、归一化和重新编号；失败则回退本地规则。

目标是理解更强，适合口语化或复杂描述。

两套生成规则怎么选

场景推荐选择原因
SAP 用例步骤标准，包含点击、输入、执行、提取、断言本地规则稳定、快、可预测，不容易合并或漏掉步骤
用例很长，但编号和格式清楚本地规则会按编号和连接词拆成细步骤

本地规则分类明显不对LLM 或手动编辑LLM 可重新规划，也可以直接改 apiGuide

菜单、Favorites、表格、滚动和 xpath

  • 在 Menu / Settings / Visualization 界面启用 Show OK Code Field 并保存 会拆成 Menu、Settings、Visualization、勾选、Save 多步。
  • 返回 SAP 主页，选择 Favorites 下面的 Asset Balances 会拆成返回主页和展开 Favorites 两步。
  • 查找 Curr.bk.val. 列的值，记录为变量 A1 会转成 aiQuery 并保存变量。
  • A1 和 A2 相等则 case 通过 会转成 aiAssert，运行时优先本地变量比较。
  • 页面拖动到最右边 会生成针对底部横向滚动条的拖动动作；cached 模式下建议 bypass 这类步骤。
  • 点击图标 xpath=//*[@id="C185_toolbar_btn10"] 会把 xpath 拆出来，直接用 DOM 定位。
  • 显式日期原样输入；动态日期如“上个月最后一个自然日”会在运行时尝试解析成 SAP 日期格式。

运行操作怎么选

操作什么时候用优势
Run raw / Run JS第一次运行、新改 T Code、新改步骤、页面结构可能变了重新规划 locator，写入新 cache
Run cached / Run JS w/ Cache用例已经成功跑过，想快速稳定复跑复用上次成功运行的定位结果
bypass cache 某一步cached 模式下某一步 locator 可能过期只让这一小步重新问模型
keep cache even if run failsraw 调试阶段，希望保留失败前已写入的定位结果下一次可复用已有进展

建议流程

  1. 先写自然语言。
  2. 点 Gen API 生成 steps。
  3. 人工快速检查 aiInput、aiQuery、aiAssert 是否合理。
  4. 第一次用 Run raw。
  5. 成功后再用 Run cached 做稳定复跑。
  6. 如果 cached 某一步失败，只勾选那一步的 bypass cache 后再跑。
