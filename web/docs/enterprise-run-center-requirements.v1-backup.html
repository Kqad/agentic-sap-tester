需求定义

这不是一个调试者 history 面板，而是给业务用户、测试负责人、自动化执行人员使用的运行控制台。

Run Orchestration

用户以项目为单位选择 case，可以运行整个项目、勾选多个 case 批量运行，也可以单独运行某一个 case。

Multi-Case Live Monitor

系统已经有单 case Live 转播能力，新的 UI 要在并发运行时为每个 case 提供清晰的 Live 入口。

Result Intelligence

每一次运行都成为完整的 run batch，记录进度、结果、截图、失败原因、报告和后续分析。

交互方案

推荐把已有 Live 功能变成运行页的一级入口：多个 case 就是多个 Live Channel。

落地路线

建议先把“能跑、能看直播、能回看结果”做稳，再补充趋势、导出、审计和智能分析。

Phase 1 · MVP

  • 项目级运行入口
  • Run Batch 数据结构
  • case 列表状态展示
  • 每个 running case 的 Live 按钮

Phase 2 · Core UX

  • Run Center 全局视图
  • Run Detail 完整详情
  • Live / Replay / Failure 状态化
  • 失败聚焦与重跑失败 case

Phase 3 · Enterprise

  • 跨项目筛选搜索
  • 报告导出与分享
  • 项目质量概览
  • 多项目并发队列监控

Phase 4 · Intelligence

  • 运行对比
  • 失败归因
  • 角色权限与审计
  • 趋势和稳定性分析

关键产品决策

这些决策会直接影响 UI 是“调试工具”还是“企业级解决方案”。

因为用户看不到后台弹出的浏览器，Live 转播承担的是“过程可信感”。UI 上要把它放在 Run Detail 和 case 行的显眼位置。
用户关注的是“一次项目执行任务”的完整结果，不是孤立 latest run。Run Batch 是后续历史、筛选、对比、报告和审计的基础。
每个 running case 都是一个 channel。用户可以在总览页面看到多个 Watch 按钮，并按需进入对应 live 转播页或面板。
失败卡片应该直接提供失败 step、截图回放、错误摘要、日志入口和重跑动作，减少用户从红色状态跳进日志里找原因的成本。
