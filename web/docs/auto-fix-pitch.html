§1 问题域与设计目标

现状

SAP WebGUI 用例目前每次挂了，典型 debug 流程是：

  1. 打开 Midscene HTML 报告，滑动帧看截图
  2. 肉眼判断是元素定位错、时序不对，还是 API 类型（tap vs keyboardPress）错了
  3. 手动打开 case JSON，改 apiGuide.steps[i].naturalLanguageInstruction / exampleCode
  4. 触发 Run raw，等 1-3 分钟看结果
  5. 大概率没修好 —— 回到第 1 步

核心痛点不是"修不出来"，而是相同类型的失败在不同 case 上反复出现，每次都从头猜。团队里已经积累的经验（Save 是双阶段弹窗、Execute 走 Enter、T-code 输入框位置）以口耳相传和代码注释的形式散落各处，新的 case 遇到时还得重新踩坑。

设计目标

  1. 降低单次 debug 的手动成本 —— agent 读报告 + 出修法 + 改 case
  2. 让团队经验可复用 —— 结构化沉淀、语义检索，不依赖记住谁在哪个 slack thread 里说过
  3. 失败也留痕 —— 无效修法也记录，避免同一坑同一 fix 反复被 LLM 提出
  4. 保守 · 敢放弃 —— 3 轮不通就叫人，不做无穷尝试
为什么不是纯 chat 助手：Chat 只能看用户手动贴的错误片段，看不到实际截图、看不到 Midscene sub-task 图；每次会话之间没有记忆；提出的修法用户还得手动搬到 case JSON 里。Auto-fix 直接与 case JSON、report、runner 打通，从"给建议"变成"能落地"。

§2 v0 已交付的能力

三条 endpoint、三个工具、一个 KB、一个客户端 loop。

Server endpoints

Endpoint行为
POST /api/case-agent/analyze 输入 { caseId, error, reportUrl }。解析 Midscene 报告拿失败 sub-task + AI thought + 截图；查 KB 拿 top-4 相关 chunks；把 KB / 截图 / thought 一起丢给 Qwen-VL；返回 { hypothesis, confidence, toolCalls: [0..1], evidence, kbHits }。
POST /api/case-agent/apply 输入 { caseId, toolCall }。改 e2e/cases/{caseId}.json 里的 apiGuide.steps[]，同时同步顶层 naturalLanguage 和 jsSource。第一次改自动落 .bak。
POST /api/case-agent/postmortem 输入 { caseId, toolCall, worked, hypothesis, ... }。算 signature 去重，写 server/data/kb/postmortems/*.md，触发增量 reindex（只 embed 新 chunk）。

Planner 可用的 tools（严格允许列表）

Tool参数覆盖的失败模式
setStepLocator { stepIdx, newLocator, reason } Locator drift · 元素找不到 / 多个匹配
insertSleepAfter { afterStepIdx, ms ≤ 5000, reason } Timing · 页面/modal 还没渲染
changeStepApi { stepIdx, newApi, newArg, reason } API mismatch · Tap vs KeyboardPress 混了
为什么就这三个 tool：v0 的选择是"覆盖 80% 的失败模式所需的最小集合"。翻检过去的 run history 和 recovery.js 里的 SAP-specific 兜底逻辑，这三类 fix 占绝大多数。splitStep / deleteStep / changeStepDelay 等有价值但目前先不加 —— 每加一个 tool，LLM 的 hypothesis 分布就要重新校准，先让当前三个跑稳再扩。

客户端能力（Case Studio Run & Debug tab）

  • 失败的 run 上多一个琥珀色 🤖 Debug 按钮
  • Modal 展示 hypothesis / confidence / 建议 fix / 可折叠 evidence 面板
  • Evidence 面板显示：失败 sub-task 类型、AI thought、时间线、KB 命中的 chunk 列表
  • Apply fix 单步 · 手动重跑 · 手动点 ✓/✗ 反馈
  • 🔁 Auto-fix (max 3)：全自动 analyze → apply → run → judge → 循环，每轮结果实时打时间线
  • signature dedup：同轮循环里 planner 提相同 fix → guard 停

KB 现状

文件
7 份 markdown（server/data/kb/general/ + .../failure-modes/ + .../postmortems/）
Chunk 数
45（按 H1/H2 切）
Embedding 模型
Qwen text-embedding-v3（DashScope）· 1024-dim
向量索引

增量策略

§3 Agent 架构与数据流

两条独立的 loop：单次 debug 的迭代 loop（同步 · 客户端驱动）+ 跨 case 沉淀的 learning loop（异步 · 每次成功/失败触发）。

Fig 1 · 上：单次 debug 的同步 loop（客户端驱动）· 下：learning 沉淀路径 · 虚线：反馈

为什么客户端驱动 debug loop

选型考虑：server-side SSE 循环也可以做，但需要处理断线重连、会话状态、并发防冲突。目前 loop 复用了已有的 /analyze、/apply、/api/midscene-js/…/run，全都是同步 endpoint。客户端只是串起来加了 iteration 计数和 signature guard。CI / 定时任务场景（v2 计划）再上服务端 loop，那时候用户不在 modal 前等，SSE 才有价值。

典型一次 auto-fix 的 wall clock

阶段典型耗时说明
/analyze（读 report + RAG + Qwen-VL）2 - 6 sQwen-VL 处理 vision 是大头
/apply（改 case JSON + 备份）< 100 ms纯本地文件 IO
run raw（Playwright + Midscene）60 - 180 s与 case 长度成正比，是循环的主耗时
/postmortem（写 md + 增量 reindex）1 - 3 s只有一个新 chunk 需要 embed

3 轮 auto-fix 最坏 ≈ 10 分钟。多数 case 在 1-2 轮内收敛或早停。

§4 KB / RAG 层设计

目标是让 planner 每次都能捞到相关经验，且新知识加进去立即可用、老知识不用重新 embed。

Chunker

server/kb/chunker.js 按 markdown 标题层级切 —— H1 提取全局标题（放进 chunk 的 title 字段），每个 H2 生成一个 chunk。单 H2 超 2400 字符时降级到 H3、再降级到段落。所有 chunk 都带 frontmatter 里的 tags / scope / tool。

// chunker output shape
{
  title:   "SAP Save / Export / Download 双阶段流程",
  section: "检测规则",
  text:    "AI thought 提到 'already showing Export as'/'Save changes dialog' → …",
  relPath: "general/sap-save-flow.md",
  meta:    { tags: ["sap","save","export","dialog","two-stage"], scope: "general", severity: "high" }
}

Embedder

server/kb/embedder.js 调 DashScope 的 text-embedding-v3（1024 维）。Batch=10（DashScope 上限）。走已有的本地 HTTP 代理 127.0.0.1:3128，跟其他 LLM 调用同一条通路，不额外要 key 和网络配置。

Embedding 输入格式故意把 heading 塞在文本前面：

function embedInputFor(chunk) {
  const heading = [chunk.title, chunk.section].filter(Boolean).join(' · ');
  return `${heading}\n\n${chunk.text}`;
}
为什么这样拼：观察到不少 chunk 正文里不重复标题词。比如"检测规则"章节讨论 Export 弹窗，但正文没写"Save"这个词。用 query "Export 弹窗保存失败" 会低分。把 heading 拼进去 embedding 就能吃到，检索质量明显好。测试过 5 个真实 query，命中率从 60% 到 100%。

Store · 单 JSON 文件 + SHA 增量

server/kb/store.js 把所有 chunk 存成一个 JSON：

{
  version: 1,
  model:   "text-embedding-v3",
  dim:     1024,
  generatedAt: "2026-07-03T14:15:00Z",
  chunks: [
    { id, title, section, text, relPath, meta, sha, embedding: [1024 floats] },
    ...
  ]
}

Reindex 时用 sha1(relPath + section + text).slice(0,12) 判断复用。真正的 embed API 调用只发生在新增或改动的 chunk 上。当前 45 个 chunk / 200KB / 一次全量 reindex 约 3 秒；增量单 chunk < 1 秒。

为什么不用 FAISS / pgvector / Weaviate：45 个 chunk 是 4 位数以内的问题；线性 cosine 是微秒级。上向量 DB 会引入新组件、新配置、新故障模式。到 5000+ chunk 或需要多进程共享索引时再迁移；Retriever 的 API 已经稳定，migration 只改一个模块。

Retriever · 三层过滤

server/kb/retriever.js 每次 query 走三层：

  1. Metadata pre-filter：按 tags / scope 缩范围。planner 已经用 hint hypothesis（Locate → locator / Tap → api-mismatch / Assert → assert）暗示。
  2. Cosine 相似度：embed query → dot product 与所有 candidate 的预归一化向量。
  3. Rare-keyword boost：正则抽 query 里 SAP T-code（AS01 / FBL1N）和 snake_case (not_found)；chunk 里出现同款关键词 +0.08，上限 +0.32。
为什么加 keyword boost：dense embedding 会把 AS01 和 VA01 拉到语义相近的位置 —— 都是 T-code。但对具体 case debug 而言，AS01 的 postmortem 跟 VA01 的往往完全不相关。简单的 exact-match 加分让检索在 domain-specific 场景里更准，同时不动 base semantic score。

Learning loop · postmortem 写回 KB

每次 auto-fix 循环里，无论 pass 还是 fail 都调 /postmortem。写盘策略：

  • Signature = sha1(caseId + stepIdx + tool + keyArgs)；重复 signature 更新同一文件（不新建）
  • Sleep 参数按 500ms 桶取整 —— 1500 和 1800 算同一签名，1500 和 3000 算不同
  • 写完立即调 chunkKbDir + reindex + refreshIndex，SHA cache 保证只有新 chunk 重新 embed（< 1s）
为什么 worked=false 也进 KB：观察到 LLM 有时候会反复提同一个失败的 fix —— 尤其在 signature 稍有变化时（比如 sleep 从 1500 → 1800）。把无效尝试也 embed 进去，配合 KB 里 "读懂 worked 字段" 章节的显式教导，planner 检索到 worked=false 记录时会主动换假设。这个负样本机制是 v0 里比较关键的一个设计。

§5 关键技术决策与取舍

每一个都可以推翻 —— 但目前的选择都有具体的项目内理由。

模型分级

用途选择为什么
Planner + Vision Qwen3-VL-Plus (DashScope) Midscene runner 本身就在用 · API key / proxy 复用 · 中文场景强 · 单次 debug 成本可控
Embedding Qwen text-embedding-v3 同一家 SDK / proxy / 计费 · 1024-dim 够用 · 中文对齐比 OpenAI 好
Escalation (planned) Opus / Sonnet 1M guard 判定卡 3 轮时兜底 · 强推理 · 1M ctx 能塞整份 KB · 单价高但触发少

Report 解析：Midscene web dump 直接 parse，不走 API

Midscene report HTML 里嵌 100+ 段 <script type="midscene_web_dump">{JSON}</script>（累积快照）+ 独立 <script type="midscene-image" data-id="uuid">dataURL</script>（截图 by uuid）。

// server/api/case-agent.js
function summariseReport(reportPath) {
  const dumps  = extractMidsceneDumps(html);       // 100+ 个累积 JSON dump
  const store  = extractScreenshotStore(html);     // uuid → dataURL map
  const execs  = mergeExecutionsFromDumps(dumps);  // 按 (execId, taskId) 保留最新
  // ... 找 firstBad · 抽 thought / subType · resolve screenshot ref
  // Assert 类无自己截图 → 自动 fallback 到前一步的截图
}
为什么直接 parse HTML 而不是让 Midscene 开个 JSON API：现有 Midscene 版本没有 dump JSON 的 endpoint，改上游有开发成本；且 report HTML 本来就落盘，parse 是纯本地 IO，无外部依赖，也不会跟着 Midscene 版本升级坏掉（dump script 的 schema 相对稳定）。如果哪天 Midscene 加了 JSON endpoint，替换 extractMidsceneDumps 的实现即可。

Tool schema 严格白名单 + 服务端 sanitize

Planner 返回的 JSON 会经过 sanitizeAnalysis：

  • hypothesis 必须在 { locator, timing, api-mismatch, data, permission, needs-human } 之内
  • toolCalls 强制取 0 或 1 个（"一次只改一处"是 KB 里也强调的约束）
  • 每个 tool 的参数按类型验证，超范围值截断（例：ms ≤ 5000、stepIdx ≤ steps.length - 1）
  • 不认识的字段直接丢弃，避免 LLM 幻觉产生奇怪 payload 传到 apply 阶段

客户端 signature dedup（避免 planner 打转）

Client 和 Server 用同款 signature 算法：

function fixSignatureClient(caseId, toolCall) {
  const a = toolCall.args;
  const stepKey = a.stepIdx ?? a.afterStepIdx ?? '?';
  let argKey = '';
  if (t === 'setStepLocator')  argKey = a.newLocator || '';
  else if (t === 'insertSleepAfter') argKey = `sleep-${Math.round((a.ms || 0) / 500) * 500}`;
  else if (t === 'changeStepApi')    argKey = `${a.newApi}::${a.newArg || ''}`;
  return `${caseId}::${stepKey}::${toolCall.tool}::${argKey}`;
}
为什么客户端也要算：服务端只在写 postmortem 时算 signature 用于文件去重，做不到"loop 内 planner 又提一遍同样的方案，立即打断"。客户端持有一个 Set<signature>，每轮 analyze 完立即检查，命中就跳到 needs-human。这是 3 轮上限之外的第二道防呆。

§6 已知风险与限制

写在这里的都是我确认存在的问题。没写的可能是我没想到 —— review 时想加请提。

R1 · Vision 成本累积：每次 /analyze 发一份 200-400KB base64 截图到 Qwen-VL。单人手动 debug 影响不大；如果哪天上 CI + 每天几百个 case，成本会显性化。Mitigation：可以给截图加分辨率上限或裁到失败元素周围；也可以对已知 hypothesis（比如 hypothesis 高置信 + tool call 明确）跳过 vision，直接走纯文本 planner。
R2 · KB cross-contamination：Postmortem 用 tag 标了 caseId，但检索不做硬过滤。saptest7 里一个"master data 按钮改新描述"的 fix，理论上可能被 saptest8 的类似 query 命中并复用 —— 而两个 case 的 SAP 界面版本可能不同。Mitigation：Retriever 加 same-caseId score boost（v0.5 计划）。
R3 · Confidence 有幻觉倾向：Qwen-VL 报出的 confidence 分布偏乐观 —— 猜错的时候也常见 0.7-0.8。当前 UI 只显示，不做门控；用户看 auto-fix 时可能会因为高置信而放松警惕。Mitigation：不打算门控 confidence（会漏正确 fix），而是在 UI 上把 evidence 面板（截图 + KB 命中）显性化 —— 让用户能快速自己 double-check。
R4 · Runner cache 污染：Auto-fix 每轮跑的是 cache=write，会往 fail slot 写 xpath。若中间某轮的 fix 是错的但恰好没让 case 整体挂（比如挂在更后面的 step），错误的 xpath 已经落进 cache。下次 Run with cache 可能踩到污染的部分。Mitigation：Auto-fix loop 开始时应该显式清一次 case 的 fail-slot cache；目前没做，是 v0.5 的 P1。
R5 · .bak 备份无清理：每次首次 apply 会在 case JSON 旁生成 .bak。目前没 rotation。Mitigation：加个"保留最近 N 份"的清理策略，或写进 server/data/case-agent/backups/<caseId>/YYYY-MM-DD-HHmmss.json。
R6 · Assert 类失败缺少直接证据：Assert task 自己不带截图，我们 fallback 到前一步的。前一步如果是 aiInput（页面还没跳转），fallback 截图对定位失败原因帮助有限。Mitigation：在 runner 里给 Assert 加一张 pre-assert 的显式截图；这动到 runner 属于 v1+ 讨论范围。
R7 · Tool 覆盖面：只有三个 tool。已经能覆盖大多数 SAP 失败，但有几类 fix 目前落不下来 ——「合并两个 step 成一个」、「删除多余 step」、「改 assert 内容（人工确认后）」。Mitigation：这些暂时只能 hypothesis: needs-human。v1 会考虑 splitStep 和 mergeStep。

§7 Roadmap

v0.5 · 客户端体验 + 关键防呆（1-2 周）

  • Retriever 加 same-caseId score boost，缓解 R2
  • Auto-fix 开始前自动清目标 case 的 fail-slot cache，解决 R4
  • Modal 里显示 KB 命中的 chunk 列表（当前只服务端能看），更容易 debug agent 本身
  • .bak 备份改成集中管理 + 保留 N 份
  • Evidence 面板加"重新分析"按钮 —— 用户可以在不 apply 的情况下再问一次

v1 · 更多 tool + 历史数据挖掘（2-4 周）

  • 加 tool：splitStep、changeQueryPreDelay、mergeSteps
  • Escalation 到 Opus / Sonnet 1M：guard 判定卡 3 轮时触发，塞入全 KB + 全 report + 全 case JSON
  • Mine 439 份历史 Midscene report + 354 份 run-history JSON：抽出 (failure signature → 后续手工 fix) 对，人审后 seed 进 postmortems/
  • KB review UI：一个 admin 面板列出所有 postmortem，可以合并、编辑、demote

v2 · 服务端 loop + CI 集成（4-6 周）

  • 把 auto-fix loop 搬到服务端，走 SSE 推进度 —— 支持 headless / cron
  • 接入 nightly regression：跑挂的 case 自动进 auto-fix 队列，第二天早上人工 review 每一份新 postmortem
  • Multi-case 并行 debug（跨 case 共享 KB 的 embedding cache）

  • Auto-fix 的默认 max iter=3 是否合适—— 我拍的，可以调

需要决策的

  • KB promotion gate：当前 worked=true 立即写进 postmortems/ 且立即可被检索。要不要加个"团队成员 3 次成功才 promote 到 general/"的 gate？

  • 失败 postmortem 的保留策略：worked=false 的记录会积累。要不要按天数轮转？还是永久保留？（永久保留的话 KB 会一直长）
